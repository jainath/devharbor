# 04 вҖ” UI Design

The aesthetic target: **calm, dense, professional dev tool**. Think Linear, Raycast, Vercel dashboard, modern VS Code вҖ” not Postman's bright UI. Monospace for paths/commands, sans for chrome.

## Window shell

- Fixed two-pane shell:
  - **Sidebar** (left, 240px, resizable 200вҖ“360): app list + dashboard link + settings.
  - **Main** (right): the currently selected view.
- Native macOS title bar with traffic lights; the sidebar starts below them. No custom window chrome in v1.
- Theme: dark by default, light available, follows system. Tailwind dark mode class strategy.

- **Each app row** (Phase 8.b): `[status dot]  [name]  [вӢ® overflow menu]`. (Inline tag chips removed Phase 15 вҖ” tags surface via tag-grouping, sidebar filter, and вҢҳP, so the row stays clean.)
  - Color dot = app identity (user-pickable hex).
  - State expressed as a colored **ring** around the dot, not a separate trailing dot. Idle = no ring; running = green; starting / exiting = amber; crashed = red. The ring lives in space the row already occupied, freeing the right slot for вӢ®.
  - **вӢ® button** is the primary trigger for the per-app menu. Hover-revealed on idle rows, always-visible on the currently selected row. Right-click on the row is preserved as a power-user fallback that opens the same menu.
- **вӢ® menu** items (same as the old right-click set, plus вӢ® is now the discoverable surface): Open В· Start/Stop В· Restart В· вҶ’ {folderвҖҰ} entries В· вҶ’ New folderвҖҰ В· вҶ’ Remove from folder В· Reveal in Finder В· Open in Terminal В· Remove from DevHarborвҖҰ
- **APPS header row** carries two action icons: **+** (Add app вҖ” same flow as before, just moved here from the bottom for discoverability) and **рҹ“Ғ+** (Create folder вҖ” opens a modal asking for a name and immediately shows the empty folder in the sidebar, ready to accept drops).
- **Group-by switcher** (Phase 15): the **APPS** header carries a Finder-style 2-segment control (Folder icon / Tag icon, active segment filled) that toggles how the list is organized. Persisted to `localStorage["devharbor:group-mode"]`.
  - **Folder mode** (default): groups by each app's single `folder`; supports create-folder (рҹ“Ғ+), folder вӢ® rename/delete, drag-reorder, and drag-app-into-folder.
  - **Tag mode**: groups by tag вҖ” an app appears under *every* tag it carries (tags are non-exclusive labels), apps with none fall into "(Untagged)" pinned last. Per-group folder affordances (вӢ® rename/delete, drag-reorder) are hidden; instead **dragging an app onto a tag group adds that tag** (non-destructive; dropping on "(Untagged)" is a no-op). With no tags anywhere, falls back to a flat list with a hint to add tags. This is the cross-folder slice: tag `api` collects every API service regardless of which folder it lives in.
  - The header action icons (group-by switcher, **+** add app, **рҹ“Ғ+** create folder) are **identical in both modes** вҖ” no layout shift when toggling. Creating a folder from tag view **auto-switches to folder view** so the new folder is visible (a folder doesn't exist in the tag grouping).
  - **App order within a folder/tag group, and in the flat list** (Phase 17) is controlled by a **sort toggle** in the header (вҮ… icon вҶ’ menu: **Name (AвҖ“Z)** В· **Recently used** В· **Running first**). This is a **single preference shared with the Dashboard** вҖ” stored as `appSort` in the renderer store (persisted to `localStorage["devharbor:app-sort"]`) so both views always agree and update live. Default **Name** is stable вҖ” it deliberately does NOT follow the DB's `updated_at DESC` (which jumps rows on start/edit). `Recently used` (by `lastStartedAt`) and `Running first` (live apps float up) are opt-ins that intentionally reorder; name is always the tiebreaker. The вҮ… icon tints accent when a non-default sort is active. Folders/tags keep their own ordering (folders: saved drag order then alphabetical; tags: alphabetical). Sort logic lives in `renderer/lib/sortApps.ts` (one comparator used by both views).
  - **The per-app вӢ® / right-click menu is view-appropriate**: in folder view its move section is `вҶ’ {folder} В· вҶ’ New folderвҖҰ В· вҶ’ Remove from folder`; in tag view it becomes `Add tag: {tag} В· New tagвҖҰ В· Remove tag: {tag}`. Everything else (Open / Start-Stop / Restart / Reveal / Terminal / Remove) is identical.
- **Folder header** (Phase 15) mirrors app rows for consistency: `[collapse caret]  NAME  [вӢ® overflow menu]`. The вӢ® is hover-revealed (right-click still works) and opens Rename folderвҖҰ В· Delete folder, both via `PromptModal` (replaces `window.prompt`, a no-op inside Electron BrowserWindow). (No per-folder count badge вҖ” removed Phase 15 as visual noise.)
- **Drag-and-drop** (Phase 8.b + 15): two drag types, distinguished by MIME so they never cross-trigger.
  - *App drag* (`application/x-appmgr-app`): an app row is `draggable`; folder sections are drop targets вҖ” drop onto a section to set its `folder`, onto "(Ungrouped)" to clear it. Hovered section gets an accent ring.
  - *Folder reorder* (`application/x-appmgr-folder`, Phase 15): a folder header is `draggable`; dropping one folder onto another reorders them. Order persists to `localStorage["devharbor:folder-order"]`; ordered folders sort first (in saved order), then unordered alphabetically, "(Ungrouped)" always last. The drop target shows a top accent border. No nested folders.
- **Folders** *(Phase 8 вҖ” F21)*: one-level grouping. Apps with `folder = NULL` collect under "(Ungrouped)". Empty folders created via рҹ“Ғ+ are stored in `localStorage["devharbor:pinned-folders"]` so they survive reload until apps are dropped in or the folder is deleted. Collapse state in `localStorage["devharbor:folder-collapse"]`.
- The Dashboard sidebar entry shows a green count badge with white text when вүҘ1 app is running.
- **Bottom row** is just the **вҡҷ Settings** icon now вҖ” Add app moved to the header.

## macOS application menu *(Phase 13)*

Native menu (`src/main/menu.ts`), following Apple HIG, with renderer-bridged actions:

- **DevHarbor menu**: About В· **SettingsвҖҰ (вҢҳ,)** вҶ’ opens the in-app Settings drawer В· Services В· Hide/Show В· Quit. вҢҳ, is the macOS-standard preferences slot.
- **File menu**: **Add AppвҖҰ (вҢҳN)** вҶ’ folder picker + add-app drawer В· **Add FolderвҖҰ (вҢҳвҮ§N)** вҶ’ new-folder modal (вҢҳвҮ§N mirrors Finder's New Folder) В· Close Window.
- **View**: Reload (вҢҳR, all builds) В· Force Reload + Toggle DevTools (dev only) В· zoom В· full-screen.
- **Help**: DevHarbor Website вҶ’ www.devharbor.app.

Menu вҶ’ renderer flow: items in the main process `webContents.send` typed events (`menu:openSettings` / `menu:addApp` / `menu:newFolder`); `App.tsx` opens the settings drawer / add-app flow directly, and relays new-folder to the sidebar via a `devharbor:new-folder` DOM event (where the folder-prompt logic lives).

## Dashboard view

The dashboard is the **control room**, deliberately distinct from the sidebar (the *switcher*). Where the sidebar navigates every app, the dashboard answers "what is my machine doing right now" and offers quick control. Layout: **aggregate stat strip** on top, **stable alphabetical card grid** below. (Phase 13: the old *Recent strip* was removed вҖ” it re-listed apps the sidebar + grid already show; recency quick-access is covered by the sidebar and вҢҳK.)

**Aggregate stat strip** вҖ” four tiles summing across all running tasks; the dashboard's unique value (the sidebar can't show this):

Each card shows one chip per task, format `<taskName> :<port>`. When the app is idle, each registered task still gets a placeholder chip (`<taskName> вҖ”`) so **the card has the same height whether running or stopped** (Phase 7 вҖ” eliminates the layout shift when clicking Start). **вҢҳ R / cold boot always lands on Dashboard** regardless of last-selected app.

## App detail view

No top-level tabs. The detail view is essentially a header + **task tabs strip (always-on, Phase 7)** + a unified Logs pane (Live or Filter mode). Env, App config are reachable via drawer buttons in the header. The header **no longer carries a "Tasks" drawer button** вҖ” the always-on strip's "Manage tasks" affordance covers that, removing the redundancy. **History was dropped from the UI** вҖ” `run_history` is still persisted to the DB, but the previous History tab didn't earn its real estate.

Two **modes** of the same task buffer, selected by an inline pill in the logs toolbar. **Phase 7 typography**: `LOGS` is the visual primary (`font-medium text-fg-muted uppercase tracking-wider`), the task name is secondary (`text-fg-subtle normal-case`), to give the header readable hierarchy instead of one flat uppercase line.

### History tab вҖ” *removed*

The History tab was dropped from the UI (Phase 7) and its `HistoryView.tsx` component deleted in the Phase 11 dead-code cleanup. Run data is still recorded to `run_history` (via `RunHistory`/`TaskRunner`) and remains readable through the `runs:list` IPC, but there is no UI surface for it today. A future "log explorer" / history view can re-add one against that data.

## Visual specifics

- **Type.** UI: system sans (`-apple-system, SF Pro, system-ui`). Code/paths/logs: `'SF Mono', 'JetBrains Mono', ui-monospace`.
- **Density.** Sidebar row height 32px. Table row height 28px. Tighter than default Tailwind.
- **Color.** Semantic CSS-variable tokens (`bg-base` / `bg-surface` / `bg-elevated` / `border` / `border-strong` / `fg` / `fg-muted` / `fg-subtle` / `accent`) driven by the **Radix Colors slate scale** for both modes вҖ” what shadcn/ui, Linear, and Vercel use. Light and dark palettes set the variable values in `.light` and `.dark` blocks; toggling `theme` in Settings flips the entire palette without per-component overrides. Status colors (`danger` / `success` / `warn`) each ship a 5-slot family (`bg` / `bg-hover` / `fg` / `border` / `strong`) calibrated for both modes. Solid mid-tone state dots (`bg-red-500` / `bg-green-500` / `bg-amber-500` / `bg-blue-500`) stay unchanged across themes вҖ” saturated mid-tones read on both backgrounds.
- **Motion.** Minimal. 120 ms ease for state pill transitions. No bouncy springs.
- **Icons.** lucide-react.
- **xterm.** Has its own dark color scheme regardless of UI theme. Intentional вҖ” matches VS Code's terminal pattern (most devs prefer dark terminals).

## Accessibility

- All actions reachable by keyboard. Visible focus rings (Tailwind `focus-visible`).
- Status conveyed by color *and* an icon/word (not color alone).
- Respect `prefers-reduced-motion`.

## What we won't do

- No marketing copy in the UI.
- No animated illustrations or onboarding tours.
- No tooltips that repeat the visible label.
- No nested-card depth gymnastics вҖ” one layer of card chrome, max.
