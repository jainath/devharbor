# 04 - UI Design

The aesthetic target: **calm, dense, professional dev tool**. Think Linear, Raycast, Vercel dashboard, modern VS Code - not Postman's bright UI. Monospace for paths/commands, sans for chrome.

## Window shell

- Fixed two-pane shell:
  - **Sidebar** (left, 240px, resizable 200-360): app list + dashboard link + settings.
  - **Main** (right): the currently selected view.
- Native macOS title bar with traffic lights; the sidebar starts below them. No custom window chrome in v1.
- Theme: dark by default, light available, follows system. Tailwind dark mode class strategy.

## Sidebar

```
┌──────────────────────────────┐
│ APP MANAGER             ⌘K   │
│ [ Search apps ⌘P         ]   │
├──────────────────────────────┤
│ ⊞ Dashboard            [1]   │
├──────────────────────────────┤
│ APPS                  [+][📁+]│  ← Phase 8.b: header actions
│ ▾ Personal               (1) │
│   ◉ jainath-com-astro    ⋮   │  ← green-ring color dot = running; ⋮ = menu
│ ▾ Work                   (2) │
│   ● auth-api             ⋮   │  ← no ring = idle
│   ● admin                ⋮   │
│ ▾ Side projects          (0) │  ← empty pinned folder
│   *Empty - drop apps here.*  │
│ ▾ (Ungrouped)            (1) │  ← reserved label, pinned to bottom
│   ● scratch              ⋮   │
├──────────────────────────────┤
│                          ⚙   │  ← settings only
└──────────────────────────────┘
```

- **Each app row** (Phase 8.b): `[status dot]  [name]  [⋮ overflow menu]`. (Inline tag chips removed Phase 15 - tags surface via tag-grouping, sidebar filter, and ⌘P, so the row stays clean.)
  - Color dot = app identity (user-pickable hex).
  - State expressed as a colored **ring** around the dot, not a separate trailing dot. Idle = no ring; running = green; starting / exiting = amber; crashed = red. The ring lives in space the row already occupied, freeing the right slot for ⋮.
  - **⋮ button** is the primary trigger for the per-app menu. Hover-revealed on idle rows, always-visible on the currently selected row. Right-click on the row is preserved as a power-user fallback that opens the same menu.
- **⋮ menu** items (same as the old right-click set, plus ⋮ is now the discoverable surface): Open · Start/Stop · Restart · → {folder…} entries · → New folder… · → Remove from folder · Reveal in Finder · Open in Terminal · Remove from DevHarbor…
- **APPS header row** carries two action icons: **+** (Add app - same flow as before, just moved here from the bottom for discoverability) and **📁+** (Create folder - opens a modal asking for a name and immediately shows the empty folder in the sidebar, ready to accept drops).
- **Group-by switcher** (Phase 15): the **APPS** header carries a Finder-style 2-segment control (Folder icon / Tag icon, active segment filled) that toggles how the list is organized. Persisted to `localStorage["devharbor:group-mode"]`.
  - **Folder mode** (default): groups by each app's single `folder`; supports create-folder (📁+), folder ⋮ rename/delete, drag-reorder, and drag-app-into-folder.
  - **Tag mode**: groups by tag - an app appears under *every* tag it carries (tags are non-exclusive labels), apps with none fall into "(Untagged)" pinned last. Per-group folder affordances (⋮ rename/delete, drag-reorder) are hidden; instead **dragging an app onto a tag group adds that tag** (non-destructive; dropping on "(Untagged)" is a no-op). With no tags anywhere, falls back to a flat list with a hint to add tags. This is the cross-folder slice: tag `api` collects every API service regardless of which folder it lives in.
  - The header action icons (group-by switcher, **+** add app, **📁+** create folder) are **identical in both modes** - no layout shift when toggling. Creating a folder from tag view **auto-switches to folder view** so the new folder is visible (a folder doesn't exist in the tag grouping).
  - **App order within a folder/tag group, and in the flat list** (Phase 17) is controlled by a **sort toggle** in the header (⇅ icon → menu: **Name (A-Z)** · **Recently used** · **Running first**). This is a **single preference shared with the Dashboard** - stored as `appSort` in the renderer store (persisted to `localStorage["devharbor:app-sort"]`) so both views always agree and update live. Default **Name** is stable - it deliberately does NOT follow the DB's `updated_at DESC` (which jumps rows on start/edit). `Recently used` (by `lastStartedAt`) and `Running first` (live apps float up) are opt-ins that intentionally reorder; name is always the tiebreaker. The ⇅ icon tints accent when a non-default sort is active. Folders/tags keep their own ordering (folders: saved drag order then alphabetical; tags: alphabetical). Sort logic lives in `renderer/lib/sortApps.ts` (one comparator used by both views).
  - **The per-app ⋮ / right-click menu is view-appropriate**: in folder view its move section is `→ {folder} · → New folder… · → Remove from folder`; in tag view it becomes `Add tag: {tag} · New tag… · Remove tag: {tag}`. Everything else (Open / Start-Stop / Restart / Reveal / Terminal / Remove) is identical.
- **Folder header** (Phase 15) mirrors app rows for consistency: `[collapse caret]  NAME  [⋮ overflow menu]`. The ⋮ is hover-revealed (right-click still works) and opens Rename folder… · Delete folder, both via `PromptModal` (replaces `window.prompt`, a no-op inside Electron BrowserWindow). (No per-folder count badge - removed Phase 15 as visual noise.)
- **Drag-and-drop** (Phase 8.b + 15): two drag types, distinguished by MIME so they never cross-trigger.
  - *App drag* (`application/x-appmgr-app`): an app row is `draggable`; folder sections are drop targets - drop onto a section to set its `folder`, onto "(Ungrouped)" to clear it. Hovered section gets an accent ring.
  - *Folder reorder* (`application/x-appmgr-folder`, Phase 15): a folder header is `draggable`; dropping one folder onto another reorders them. Order persists to `localStorage["devharbor:folder-order"]`; ordered folders sort first (in saved order), then unordered alphabetically, "(Ungrouped)" always last. The drop target shows a top accent border. No nested folders.
- **Folders** *(Phase 8 - F21)*: one-level grouping. Apps with `folder = NULL` collect under "(Ungrouped)". Empty folders created via 📁+ are stored in `localStorage["devharbor:pinned-folders"]` so they survive reload until apps are dropped in or the folder is deleted. Collapse state in `localStorage["devharbor:folder-collapse"]`.
- The Dashboard sidebar entry shows a green count badge with white text when ≥1 app is running.
- **Bottom row** is just the **⚙ Settings** icon now - Add app moved to the header.

## macOS application menu *(Phase 13)*

Native menu (`src/main/menu.ts`), following Apple HIG, with renderer-bridged actions:

- **DevHarbor menu**: About · **Settings… (⌘,)** → opens the in-app Settings drawer · Services · Hide/Show · Quit. ⌘, is the macOS-standard preferences slot.
- **File menu**: **Add App… (⌘N)** → folder picker + add-app drawer · **Add Folder… (⌘⇧N)** → new-folder modal (⌘⇧N mirrors Finder's New Folder) · Close Window.
- **View**: Reload (⌘R, all builds) · Force Reload + Toggle DevTools (dev only) · zoom · full-screen.
- **Help**: DevHarbor Website → www.devharbor.app.

Menu → renderer flow: items in the main process `webContents.send` typed events (`menu:openSettings` / `menu:addApp` / `menu:newFolder`); `App.tsx` opens the settings drawer / add-app flow directly, and relays new-folder to the sidebar via a `devharbor:new-folder` DOM event (where the folder-prompt logic lives).

## Dashboard view

The dashboard is the **control room**, deliberately distinct from the sidebar (the *switcher*). Where the sidebar navigates every app, the dashboard answers "what is my machine doing right now" and offers quick control. Layout: **aggregate stat strip** on top, **stable alphabetical card grid** below. (Phase 13: the old *Recent strip* was removed - it re-listed apps the sidebar + grid already show; recency quick-access is covered by the sidebar and ⌘K.)

**Aggregate stat strip** - four tiles summing across all running tasks; the dashboard's unique value (the sidebar can't show this):

```
┌──────────┬──────────┬───────────┬─────────────┐
│ RUNNING  │ CPU      │ MEMORY    │ OPEN PORTS  │
│ 2 / 5    │ 14.2%    │ 412 MB    │ 3           │
└──────────┴──────────┴───────────┴─────────────┘
```

The RUNNING value is teal when > 0. Memory rolls up to GB past 1024 MB. Ports = count of detected listening ports across running tasks.

**All apps grid** - every registered app, ordered by the **shared sort** (⇅ in the header: Name · Recently used · Running first - the *same* `appSort` store value the Sidebar uses, so the two views always agree and sync live). Default **Name** is stable: running apps are emphasised **in place** - a teal inset left edge + lifted surface - never reordered (click targets stay put) and with no layout shift (inset box-shadow). (Picking *Running first* intentionally floats live apps to the top - handy for the control room.) Example card:

```
┌─────────────────────────────────┐
│ ● auth-api                RUNNING│
│ 2 tasks · 1 running              │
│ CPU 12%   MEM 184 MB    ▁▂▃▄▅▃▂  │
│ [ migrate ✓ ] [ api :4000 ]      │   ← port row: reserved height
│ [ web :3000 ]                    │
│ [Stop]  [Logs]                   │
└─────────────────────────────────┘
```

Each card shows one chip per task, format `<taskName> :<port>`. When the app is idle, each registered task still gets a placeholder chip (`<taskName> - `) so **the card has the same height whether running or stopped** (Phase 7 - eliminates the layout shift when clicking Start). Up to 6 chips, then `+N` overflow. The card itself has a `min-h` constraint so cards in one row stay aligned even when only one of them is running.

**Top toolbar**: `[Filter by tag ▾]  [Stop all]`. (Start-all removed - too easy to misfire.)

**Sidebar Dashboard entry** shows a green count badge (`success-strong` bg, `bg-base` fg) when ≥1 app is running.

**⌘ R / cold boot always lands on Dashboard** regardless of last-selected app.

## App detail view

No top-level tabs. The detail view is essentially a header + **task tabs strip (always-on, Phase 7)** + a unified Logs pane (Live or Filter mode). Env, App config are reachable via drawer buttons in the header. The header **no longer carries a "Tasks" drawer button** - the always-on strip's "Manage tasks" affordance covers that, removing the redundancy. **History was dropped from the UI** - `run_history` is still persisted to the DB, but the previous History tab didn't earn its real estate.

### Header (always visible)

```
● auth-api  ● running · 3m            [Open in ▾]  [Env]  [⚙]  [Start (⌘ ↩)]  [🗑]
~/code/auth-api  [⧉]  · pnpm · node 20.11.0      ← copy-path icon next to path
[ migrate ✓ ]  [ api :4000 ]  [ web :3000 ]      ← own row, **fixed height even when idle**
```

- State pill (green / amber / red / blue) - **app-level**, derived from all task states.
- ~~Script picker dropdown - removed Phase 7 round 2.~~ Single-task and multi-task apps now share the **identical** header. Editing a task's script lives in the task tab right-click → Edit, consistent across task counts.
- **Open in** menu: Finder, Terminal, VS Code, Cursor, Sublime (auto-detected via `which`).
- **Env**: opens EnvEditor drawer (defaults to App scope; switch to Task scope from the editor's tab bar - Phase 7).
- **⚙**: opens AppConfigDrawer (name / tags / folder / auto-restart / watch globs / **Node version**). Color picker removed (Phase 14); Open in detects via bundle existence + LaunchServices, launched with `open -a` (Phase 14, not `which`).
- Start / Stop / Restart act on the whole task graph. All three lifecycle buttons share `min-w-[88px]` so they're visually the same size regardless of label.
- **Trash icon (smart delete, Phase 7 round 2)**: if the app is running, the confirm dialog asks "Stop all tasks and remove?" - clicking Yes stops the orchestrated graph then removes the app. If not running, it just confirms removal. Removes only from DevHarbor's registry; **never deletes files**.
- **Path row with copy icon (Phase 7)**: `~/code/auth-api  [⧉]`. Click the icon → `navigator.clipboard.writeText(app.path)`, brief checkmark flash.
- **Port chip row (always rendered, Phase 7)**: own row, fixed height (`min-h-[26px]`). When running: one chip per task. When idle: one muted placeholder chip per registered task at the same height. The header height is constant across all state transitions - no reflow when the user clicks Start.

### Task tabs *(Phase 1.5; always-on since Phase 7)*

A horizontal strip directly below the header, above the log pane - **rendered for both single-task and multi-task apps**. Single-task apps show one tab plus the Manage button. Reason: consistency, no special-cased UI, and right-click affordances on tasks (Start / Stop / Restart this task / Disable / Edit / Remove) become available regardless of task count.

```
 ┌──────────────────────────────────────────────────────────────────────────┐
 │ ● migrate ✓   ● api :4000   ● web :3000   + add task     [Manage tasks]  │
 └──────────────────────────────────────────────────────────────────────────┘
```

- Each tab = one Task. Dot color = task state (green running+ready, amber starting/running-not-ready, blue oneShot-completed, red crashed, grey idle/disabled).
- Tab content: task name + readiness hint (`:4000` for port, `✓` for completed oneShot, `…` while waiting on deps).
- Click a tab → that task's xterm fills the log pane.
- Hover → tooltip: command, working dir, dependsOn, readiness signal.
- Right-click → per-task menu: Start / Stop / Restart this task only, Disable, Edit, **Edit task env…** (Phase 7), Remove.
- "+ add task" opens the task editor in "new task" mode.
- Right-edge **Manage tasks** button (replaces the header drawer button).

### Logs pane (the only pane)

Two **modes** of the same task buffer, selected by an inline pill in the logs toolbar. **Phase 7 typography**: `LOGS` is the visual primary (`font-medium text-fg-muted uppercase tracking-wider`), the task name is secondary (`text-fg-subtle normal-case`), to give the header readable hierarchy instead of one flat uppercase line.

```
LOGS · dev-web    [ Live | Filter ]                          [Copy] [Save] [Clear]
─────────────────────────────────────────────────────────────────────────────────
xterm  OR  virtualized regex filter list
```

- **Live** (default): xterm filling the pane, WebGL renderer, full ANSI / TUI fidelity. ⌘ F opens a small floating search overlay (step-through highlight). Jump-to-bottom chip appears when scrolled away. Font ± buttons in the toolbar.
- **Filter**: virtualized list (`react-window` + `anser`) showing only matching lines, with regex / case-sensitive toggles, line numbers, match count.

The right-side toolbar (Copy / Save / Clear) is visible in Live only - those operations don't apply to a filtered view (the in-pane search has its own controls).

**Crash pin** (above the pane, Live mode only) - when the active task is in `crashed` state, a sticky red strip shows the last 200 lines + `[Restart task]` + dismiss. Auto-resets on task switch and on the next crash.

### History tab - *removed*

The History tab was dropped from the UI (Phase 7) and its `HistoryView.tsx` component deleted in the Phase 11 dead-code cleanup. Run data is still recorded to `run_history` (via `RunHistory`/`TaskRunner`) and remains readable through the `runs:list` IPC, but there is no UI surface for it today. A future "log explorer" / history view can re-add one against that data.

### Env editor *(drawer, three scopes since Phase 7)*

Tabs: **Global** · **App · {name}** · **Task · {name}** (third tab only present when the editor was opened with a task context).

```
┌──────────────────────────────────────────────────────────────────────┐
│ [ Global | App · auth-api | Task · api ]    [Paste .env] [Export]    │
├─┬────────────────────────┬──────────────────────┬────────────────────┤
│☑│ KEY                    │ VALUE                │ NOTE               │
├─┼────────────────────────┼──────────────────────┼────────────────────┤
│☑│ PORT                   │ 4000                 │                    │
│☑│ DATABASE_URL  🔒       │ ••••••••  ⤴ overrides app │              │
│☑│ DEBUG                  │ app:auth:trace  ⤴ overrides global │     │
└─┴────────────────────────┴──────────────────────┴────────────────────┘
```

- Tab order intentionally reads bottom-up: Global is the floor, App overrides Global, Task overrides App. Override chips on rows make this explicit.
- Right-side panel: "Effective merged env (read-only) - grouped by winning scope" - sections in order **Task → App → Global**, each showing only the keys where that scope is the winner (shadowed values are intentionally omitted from the preview; the editable table is where you go to see/change them).
- Sub-panel: ".env files found" - shows discovered files with last-modified.
- The Task tab is opened from:
  - **Task strip right-click → "Edit task env…"**
  - **TaskEditor → Env section** (`Open task env in editor`)
  - **From within the Env editor**, when an App-level row's task counterpart is being authored, a small "to Task ·" link.

### Config tab

Two sub-sections: **App** (top) and **Tasks** (bottom).

**App** (form layout - implemented as `AppConfigDrawer` accessed via the ⚙ button):

- Name
- **Tags** (Phase 15) - `TagInput`: a chip editor with autocomplete. Type + Enter/comma adds a chip, Backspace on empty input removes the last, a dropdown suggests tags already used on other apps (↑/↓ to navigate, Esc to dismiss). Tags drive: sidebar search match + the two tags shown on each app row, the Dashboard "Filter by tag" dropdown, and the ⌘P switcher (typing a tag surfaces matching apps, which also display their tags). One free-form label set per app - the way to slice across folders.
- **Folder** (Phase 15) - `FolderSelect` combobox replacing the old plain text+datalist: a button shows the current folder (or "No folder"); the dropdown lists "No folder" + every existing folder (checkmark on the current) and a "Create new folder…" action that reveals an inline name input. One folder per app.
- **Node version**: `Auto (.nvmrc / .node-version / engines.node) ▾` | `System Node` | explicit installed version (deduped across nvm/fnm/volta/asdf/system, sources shown)
- Auto-restart on file change (toggle + glob list, default `src/**/*.{ts,tsx,js,jsx,mjs,cjs}`)
- (Working directory, package manager override - schema supports; UI fields deferred)
- (Save logs to disk, kill-grace override - schema supports; UI fields deferred)

Removal lives at the trash icon in the App Detail header, not the config drawer.

**Tasks** *(Phase 1.5)*:

A table of tasks, drag-handle on the left for reordering:

```
┌──┬──────────┬──────────────────────┬──────────┬──────────────┬─────────┐
│⋮⋮│ NAME     │ COMMAND              │ READY    │ DEPENDS ON   │ ⋯       │
├──┼──────────┼──────────────────────┼──────────┼──────────────┼─────────┤
│⋮⋮│ migrate  │ pnpm -F api migrate  │ exit 0   │ -            │ ☑ enabled│
│⋮⋮│ api      │ npm run dev          │ port:4000│ migrate      │ ☑       │
│⋮⋮│ web      │ npm run dev (apps/web)│ port:3000│ api          │ ☑       │
└──┴──────────┴──────────────────────┴──────────┴──────────────┴─────────┘
[+ Add task]
```

Inline edit per row, or click a row → side panel with full editor:

- Name
- Command kind: script (pick from package.json) or custom (raw shell line)
- Working dir override (defaults to App's; lets one task target `apps/api`)
- Package manager / Node version overrides (collapsed by default)
- `dependsOn` - multi-select dropdown of sibling tasks
- Readiness: radio of (none / port / log / exit / delay) + the parameter input
- One-shot toggle (forces readiness=exit)
- Per-task env overrides (small KEY=VALUE editor; full env layering UI lives in the Env tab)
- Enabled toggle

Save-time validation: cycle detection on `dependsOn`, one-shot must have `readiness=exit`, names unique within the app.

## Add-app flow

A **single, progressively-disclosed screen** (`AddAppDrawer`), not a wizard. You browse for a
folder first; the rest reveals once a valid, not-already-registered folder is detected:

```
┌─ Add app ──────────────────────────────────────────┐
│  ① Project folder                                  │
│     [ 📁 ~/code/auth-api               Change ]     │
│     (inline error if already added; "Scan this     │
│      folder" gesture for deep-link paths)          │
│                                                    │
│  What we found:  Scripts 4 · pnpm · .env, .env.local│
│  (warns inline if no package.json found)           │
│                                                    │
│  ② How it runs                                     │
│     Name            [auth-api          ]  (max 60) │
│     Node version    [Auto ▾]  (follows .nvmrc pin) │
│     Package manager [pnpm (detected) ▾]            │
│     Start script    [dev ▾] → becomes first task   │
│     ▸ Monorepo detected - N workspace packages     │
│       [ ] Create a task per workspace package      │
│                                                    │
│  ③ Environment variables (optional, collapsed)     │
│     ▸ Paste a .env blob                             │
│                                                    │
│                       [Cancel]   [Add app]         │
└────────────────────────────────────────────────────┘
```

Tags are **not** in this drawer (set later in App settings). The whole add is **one atomic
`apps:create` IPC** - main commits the app + first task (or one task per workspace) + env vars
in a single transaction with rollback, so a thrown step never leaves an orphan app row. The
drawer is an accessible dialog (focus trap / Escape / focus restore via `useDialog`).

Bulk import: **"Import projects…"** (EmptyState, sidebar `+` menu, ⌘K) shallow-scans a chosen
folder for `package.json` projects and batch-registers the selected ones via `apps:create`.

## Visual specifics

- **Type.** UI: system sans (`-apple-system, SF Pro, system-ui`). Code/paths/logs: `'SF Mono', 'JetBrains Mono', ui-monospace`.
- **Density.** Sidebar row height 32px. Table row height 28px. Tighter than default Tailwind.
- **Color.** Semantic CSS-variable tokens (`bg-base` / `bg-surface` / `bg-elevated` / `border` / `border-strong` / `fg` / `fg-muted` / `fg-subtle` / `accent`) driven by the **Radix Colors slate scale** for both modes - what shadcn/ui, Linear, and Vercel use. Light and dark palettes set the variable values in `.light` and `.dark` blocks; toggling `theme` in Settings flips the entire palette without per-component overrides. Status colors (`danger` / `success` / `warn`) each ship a 5-slot family (`bg` / `bg-hover` / `fg` / `border` / `strong`) calibrated for both modes. Solid mid-tone state dots (`bg-red-500` / `bg-green-500` / `bg-amber-500` / `bg-blue-500`) stay unchanged across themes - saturated mid-tones read on both backgrounds.
- **Motion.** Minimal. 120 ms ease for state pill transitions. No bouncy springs.
- **Icons.** lucide-react.
- **xterm.** Has its own dark color scheme regardless of UI theme. Intentional - matches VS Code's terminal pattern (most devs prefer dark terminals).

## Accessibility

- All actions reachable by keyboard. Visible focus rings (Tailwind `focus-visible`).
- Status conveyed by color *and* an icon/word (not color alone).
- Respect `prefers-reduced-motion`.

## What we won't do

- No marketing copy in the UI.
- No animated illustrations or onboarding tours.
- No tooltips that repeat the visible label.
- No nested-card depth gymnastics - one layer of card chrome, max.
